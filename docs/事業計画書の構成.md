# 事業計画書の構成

この文書は、Claude Code 契約者が `/draft-section` で各セクションを作る前に、事業計画書全体の役割分担を理解するためのガイドです。`templates/事業計画書テンプレ.md` と同じ標準セクションを扱い、各セクションについて「目的」「補助金審査での見られ方」「書く順番」「数値根拠の置き方」「ありがちな減点」を整理します。

作成者は顧客本人です。AI は材料整理、構成検討、文章案の壁打ちを補助するだけであり、行政書士法に抵触する申請書の作成代行、代理提出、本人に代わる完成判断は行いません。公式の募集要項、様式、審査項目、文字数指定がこの文書と違う場合は、必ず募集要項を優先してください。数値や要件を確認できない箇所には `[要確認]` を付けます。

## 使うタイミング

次の場面で参照してください。

- `/draft-section` でどのセクションから書くか迷っている。
- 事業概要、現状課題、事業内容、実施体制、スケジュール、資金計画、効果・KPI のつながりを確認したい。
- 文章は書けているが、補助金審査でどこを見られるか不安がある。
- 売上、従業員数、投資額、補助率、補助上限、効果見込み、KPI などの数値根拠をどこに置くべきか整理したい。
- `/review` に渡す前に、根拠不足や誇張を自分で点検したい。

## 全体設計

補助金申請用の事業計画書は、単に会社紹介や投資説明を並べる文書ではありません。審査では、募集要項の目的に対して「なぜ必要か」「何を実施するか」「本当に実行できるか」「どのような効果があるか」「その説明に根拠があるか」が見られます。

基本の流れは次の順です。

1. 事業概要で、申請者が対象者として妥当かを示す。
2. 現状課題で、補助事業が必要な理由を示す。
3. 事業内容で、課題への解決策と対象経費を示す。
4. 実施体制で、計画を実行できる人員、経験、協力先を示す。
5. スケジュールで、募集要項の期間内に実施できる見通しを示す。
6. 資金計画で、経費、自己負担、支払、補助金入金までの現実性を示す。
7. 効果・KPI で、補助事業後に何が改善し、どう測るかを示す。

この順番は、`/draft-section` で 1 セクションずつ作る場合の考え方です。実際の様式で順番や見出しが指定されている場合は、様式を優先してください。

## セクション別ガイド

### 事業概要

#### 目的

事業概要は、申請者がどのような事業者で、なぜこの補助金の対象者として説明できるのかを示すセクションです。会社紹介を広く書く場所ではなく、補助事業につながる事業内容、顧客、商圏、沿革、強み、既存実績を整理します。

#### 補助金審査での見られ方

審査では、申請者の実態が募集要項の対象者要件と合っているか、補助事業を実施する土台があるかが見られます。所在地、業種、規模、法人形態、創業時期、従業員数、売上規模などが要件に関係する場合は、募集要項の定義と自社資料を照合してください。

#### 書く順番

1. 会社名、事業内容、主な商品・サービスを簡潔に書く。
2. 主な顧客、商圏、提供価値を示す。
3. 沿革、実績、強みのうち、補助事業に関係するものを選ぶ。
4. 募集要項の対象者要件に関係する情報を根拠付きで置く。
5. 補助事業で解決したい課題へ自然につなぐ。

#### 数値根拠の置き方

売上、従業員数、顧客数、店舗数、生産量、取引件数などは、会社資料や決算資料で確認できる範囲に限定します。対象者要件に関係する数値は、募集要項の定義と測定時点を確認し、未確認なら `[要確認]` としてください。規模を大きく見せるための推測値は使いません。

#### ありがちな減点

- 会社紹介が長く、補助事業との関係が薄い。
- 対象者要件に関係する情報が抜けている。
- 売上規模、従業員数、所在地などを根拠なしに断定している。
- 強みだけを書き、次の現状課題につながっていない。

### 現状課題

#### 目的

現状課題は、なぜ補助事業が必要なのかを説明するセクションです。一般的な業界課題ではなく、申請者本人の事業で実際に起きている制約、機会損失、非効率、品質課題、販路課題などを明確にします。

#### 補助金審査での見られ方

審査では、課題が補助金の政策目的や審査項目とつながっているか、補助事業で解決すべき必要性があるかが見られます。課題が弱いと、事業内容や効果が良く見えても「なぜ補助が必要か」が伝わりません。

#### 書く順番

1. 主要な課題を 1 つから 3 つに絞る。
2. 課題が起きている現場、業務、顧客接点を説明する。
3. 課題の原因や背景を、自社資料で確認できる範囲で示す。
4. 課題を放置した場合の影響を書く。
5. 次の事業内容で何を解決するのかを予告する。

#### 数値根拠の置き方

売上減少、作業時間、廃棄率、受注機会、顧客数、問い合わせ数、設備稼働率など、課題を示す数値は本文の近くに置きます。数値がない場合は、現場記録、顧客の声、写真、業務メモなど、顧客本人が確認できる根拠を示します。根拠がないまま「深刻」「大幅」「著しい」と表現する箇所は `[要確認]` にします。

#### ありがちな減点

- どの会社にも当てはまる一般論だけになっている。
- 課題が多すぎて、補助事業で何を解決するのか不明確。
- 課題と事業内容がつながっていない。
- 根拠資料がない強い表現を使っている。

### 事業内容

#### 目的

事業内容は、補助事業で何を導入、開発、改善、実施するのかを説明する中心セクションです。設備、システム、外注、広告、開発、改装などの内容を、現状課題への解決策として整理します。

#### 補助金審査での見られ方

審査では、実施内容が募集要項の対象事業や対象経費に合っているか、課題解決に必要か、政策目的や加点要素とつながっているかが見られます。ここで対象外経費や事業目的との不整合があると、計画全体の評価に影響します。

#### 書く順番

1. 導入または実施する内容を具体的に書く。
2. それが現状課題をどう解決するかを説明する。
3. 対象経費区分、対象事業、政策目的との対応を示す。
4. 導入後の業務フローや顧客提供価値の変化を書く。
5. 不足準備や確認事項を `[要確認]` として残す。

#### 数値根拠の置き方

投資額、見積額、仕様、数量、対象経費区分、補助率、補助上限は、見積書と募集要項を根拠にします。効果見込みにつながる数値は、事業内容の説明だけで完結させず、効果・KPI セクションと対応させます。対象経費かどうか不明な項目は断定せず `[要確認]` としてください。

#### ありがちな減点

- 欲しい設備やサービスの説明だけで、課題解決との関係が薄い。
- 対象経費かどうかを募集要項で確認していない。
- 導入後の業務や顧客への変化が書かれていない。
- 補助金に合わせて事業内容を誇張している。

### 実施体制

#### 目的

実施体制は、計画を実行できる人員、役割、経験、外部協力先、支援機関、許認可などを示すセクションです。良い事業内容でも、誰が責任を持って進めるかが不明だと実現性が弱く見えます。

#### 補助金審査での見られ方

審査では、代表者や担当者の関与、社内の役割分担、外部協力先の妥当性、過去実績、必要な資格や許認可が確認されます。補助事業の規模に対して体制が過小または不明確だと、実行可能性に疑問が残ります。

#### 書く順番

1. 責任者と主担当者を示す。
2. 社内の役割分担を整理する。
3. 外部協力先、ベンダー、支援機関の役割を示す。
4. 過去実績、資格、許認可、類似経験を根拠として置く。
5. 未確定の担当、契約先、許認可は `[要確認]` とする。

#### 数値根拠の置き方

担当人数、稼働時間、過去実績件数、導入経験、資格保有者数など、体制の実現性を支える数値を使います。ただし、人員計画を実態より大きく見せることは避けます。外部協力先については、見積書、提案書、契約予定、支援機関の確認記録などを根拠にします。

#### ありがちな減点

- 担当者名や役割がなく、誰が実行するか不明。
- 外部委託先に丸投げしているように見える。
- 必要な資格、許認可、経験の確認が抜けている。
- 体制を強く見せるために未確定事項を断定している。

### スケジュール

#### 目的

スケジュールは、補助事業を募集要項の期間内に正しい順序で実施できることを示すセクションです。申請、採択、契約、発注、納品、支払、実績報告、効果測定の流れを整理します。

#### 補助金審査での見られ方

審査では、事業実施期間に収まっているか、採択前の契約や支払など対象外になり得る行為がないか、必要な準備期間が確保されているかが見られます。無理な工程は、計画の実現性を弱めます。

#### 書く順番

1. 募集要項の申請締切と事業実施期間を確認する。
2. 採択前に行ってよいこと、採択後に行うことを分ける。
3. 契約、発注、納品、検収、支払、実績報告を順に並べる。
4. 社内準備、導入、運用開始、効果測定の時期を置く。
5. 日付や期間が未確認なら `[要確認]` とする。

#### 数値根拠の置き方

日付、期間、発注から納品までの日数、支払予定日、効果測定時期は、募集要項、見積書、社内予定、ベンダー回答を根拠にします。補助金では、契約や支払のタイミングが対象可否に関係することがあります。日付を仮置きする場合も、仮置きであることを明示してください。

#### ありがちな減点

- 事業実施期間の確認がなく、日付が募集要項と合っていない。
- 採択前に契約、発注、支払をする前提になっている可能性がある。
- 納品、支払、実績報告、効果測定の順序が曖昧。
- 短期間に無理な工程を詰め込んでいる。

### 資金計画

#### 目的

資金計画は、補助事業に必要な経費、補助対象と対象外、自己負担、支払時期、補助金入金までの資金繰りを示すセクションです。補助金は後払いになることが多いため、支払を実行できる現実性が重要です。

#### 補助金審査での見られ方

審査では、経費が募集要項の対象経費に合っているか、金額が見積書と一致しているか、補助率や補助上限の理解が正しいか、自己負担を用意できるかが見られます。資金計画が曖昧だと、事業実施の実現性が疑われます。

#### 書く順番

1. 見積書に基づいて経費項目と金額を分ける。
2. 募集要項の対象経費区分と照合する。
3. 対象外経費、消費税、自己負担を区別する。
4. 補助率、補助上限、補助予定額を募集要項で確認する。
5. 支払時期と資金調達の見通しを示す。

#### 数値根拠の置き方

金額は見積書、仕様書、資金繰り表、金融機関資料などに基づきます。補助率、補助上限、補助対象経費の定義は募集要項を根拠にし、一般的な補助金知識で推測しません。自己負担額や補助予定額を計算する場合は、計算式と根拠資料を併記し、未確認なら `[要確認]` としてください。

#### ありがちな減点

- 見積書と計画書の金額が一致していない。
- 対象経費と対象外経費が混ざっている。
- 補助率、補助上限、消費税、自己負担を推測している。
- 補助金入金までの支払原資が説明されていない。

### 効果・KPI

#### 目的

効果・KPI は、補助事業の成果をどの指標で測り、どの根拠で見込みを立てるかを示すセクションです。売上増加、生産性向上、作業時間削減、品質向上、販路拡大、雇用、地域波及などを、募集要項の審査項目に合わせて整理します。

#### 補助金審査での見られ方

審査では、効果が政策目的や加点要素とつながっているか、現状値と目標値が根拠に基づいているか、測定方法が現実的かが見られます。効果が大きくても根拠が薄い場合、誇張と見られる可能性があります。

#### 書く順番

1. 補助事業で改善する指標を選ぶ。
2. 現状値、目標値、測定時期、測定方法を整理する。
3. 目標値の根拠となる既存実績、見積、商談、試算を示す。
4. 審査項目、政策目的、加点要素との対応を書く。
5. 根拠が薄い効果は断定せず `[要確認]` とする。

#### 数値根拠の置き方

KPI は、現状値、目標値、算出式、測定方法をセットにします。たとえば、作業時間削減なら現状の作業時間、導入後の想定時間、根拠となる仕様や実績を示します。売上増加なら、既存売上、商談数、単価、販売数量、販路計画など、顧客本人が説明できる根拠を置きます。根拠のない成長率や採択されやすそうな数値は使いません。

#### ありがちな減点

- 効果が抽象的で、測定指標がない。
- 現状値、目標値、測定方法がそろっていない。
- 根拠のない売上増加や生産性向上を断定している。
- 募集要項の審査項目や加点要素と効果がつながっていない。

## 書く順番の実務メモ

最初から全セクションを完成させようとせず、`/draft-section` で 1 セクションずつ叩き台にしてください。迷う場合は、現状課題、事業内容、効果・KPI の 3 つを先に粗く作ると、補助事業の必要性と成果の線が見えやすくなります。その後、事業概要、実施体制、スケジュール、資金計画で実現性と根拠を補います。

ただし、募集要項の様式が先に決まっている場合は、様式の見出しに合わせてください。この文書の標準セクション名を無理にそのまま使う必要はありません。

## `/review` 前の確認

`/review` に渡す前に、次を確認してください。

- 各セクションが募集要項の見出し、審査項目、文字数指定に対応している。
- 事業概要、現状課題、事業内容、実施体制、スケジュール、資金計画、効果・KPI の説明が互いに矛盾していない。
- 数値根拠のない主張、出典不明の事実、顧客確認が必要な内容に `[要確認]` が残っている。
- 強すぎる表現、採択可能性の断定、補助金額の断定、根拠のない効果見込みがない。
- 作成者は顧客本人であり、AI が完成判断や提出判断をしている表現になっていない。

最後に、公式募集要項の最新版と自社資料を顧客本人が確認してください。この文書は構成の地図であり、申請内容の確定や提出判断を代行するものではありません。
