# ハーネスの育て方

この文書は、saita-kun-planner を clone した顧客本人が、自分の補助金申請環境を壊さず育てるための運用ガイドです。ここでいうハーネスは、Claude Code の slash command、spec、検証ツール、マニュアル、テンプレートをまとめた作業基盤を指します。

申請書の作成者は顧客本人です。AI は補助、壁打ち、整理役であり、申請代行、代理提出、本人に代わる完成判断は行いません。数値、要件、文字数、補助率、補助上限、提出期限、添付資料は公式の募集要項または顧客資料で確認し、不明なものは `[要確認]` として残してください。

## 二層モデル

この repo は、コア層と育成層を分けて運用します。

| 層 | 役割 | 主な場所 | 更新時の扱い |
| --- | --- | --- | --- |
| コア層 | テンプレート提供側が改善する共通基盤 | `core-manifest.json` に載っている `.claude/commands/`、`docs/`、`schemas/`、`specs/`、`templates/`、`tools/` など | `tools/update-core.sh` の対象 |
| 育成層 | 顧客本人の申請情報、学び、固有運用 | `input/`、`knowledge/`、`.claude/commands/my-*.md` | コア更新では触れない |

`core-manifest.json` は、どのファイルをコア層として扱うかの正本です。ここに載っているファイルは、上流テンプレートが改善したときに更新候補になります。逆に、`input/`、`knowledge/`、`.claude/commands/my-*.md` は顧客本人の資産として残す前提です。

補助金パックも同じ考え方で分けます。同梱パックは `specs/<subsidy_id>/` に置かれるコア層で、上流更新の対象です。顧客本人が `/ingest-guidelines`、`/confirm-spec`、`/build-pack` で作る自作パックは `input/spec/<subsidy_id>/` に置く育成層で、会社固有の狙い、確認履歴、書き方メモが混ざるため既定ではコミットしません。

## コア層に含まれるもの

コア層は、補助金申請の作業を成立させる共通部品です。

- `.claude/commands/` の標準コマンド: `/setup`、`/start`、`/select-subsidy`、`/ingest-guidelines`、`/confirm-spec`、`/build-pack`、`/intake`、`/subsidy-fit`、`/plan-deliverables`、`/draft-section`、`/review`、`/verify`、`/finalize`、`/retrospect`
- `schemas/`: company profile、subsidy spec、application record などの構造定義
- `specs/`: 提供側が確認済みとして同梱する bundled spec
- `tools/`: `check-spec.sh`、`check-drafts.sh`、`update-core.sh` などの検証・更新ツール
- `docs/` と `templates/`: 利用者向けの説明、FAQ、作業テンプレート
- `README.md`、`CLAUDE.md`、`TERMS.md`、`core-manifest.json`

これらは使い方の改善や spec 形式の進化に合わせて上流から取り込む可能性があります。顧客固有の会社情報、募集要項メモ、申請結果、社内手順はコア層へ直接書かないでください。

## 育成層に含まれるもの

育成層は、顧客本人の repo で育てる資産です。

- `input/`: 今回の会社プロフィール、募集要項原文、`input/spec/`、`input/current-application.json`、draft、review、verify report、提出前チェック
- `input/spec/<subsidy_id>/`: 顧客本人が作る自作パック。confirmed spec、confirmation、書き方メモを置く場所
- `knowledge/`: `/retrospect` で残す申請結果、講評、フェーズ別の学び、次回アクション
- `.claude/commands/my-*.md`: 顧客本人または支援者が追加する独自コマンド

`input/` は機密情報を含みやすいため `.gitignore` されています。`knowledge/` は次回申請に活かす長期資産としてコミットする前提で、会社名、取引先名、金額、講評などを含むため、作業 repo は必ず private にしてください。

## 補助金パックの置き場所

| 種類 | 置き場所 | 層 | 扱い |
| --- | --- | --- | --- |
| 同梱パック | `specs/<subsidy_id>/` | コア層 | 提供側が確認済みとして配る。上流更新で改善される |
| 自作パック | `input/spec/<subsidy_id>/` | 育成層 | 顧客本人の募集要項から作る。既定ではコミットしない |
| 旧平置き spec | `input/spec/<subsidy_id>.json` や `specs/<subsidy_id>.json` | 互換用 | パック形がある場合はパック形を優先する |

`/select-subsidy` は同梱パックを選ぶ入口です。`/ingest-guidelines` は公式募集要項を `input/guidelines/` から読み込んで draft spec を作り、`/confirm-spec` は顧客本人が募集要項と突合確認します。`/build-pack` は confirmed spec から書き方メモを作り、自作パックを `input/spec/<subsidy_id>/` に整えます。

コア更新後、上流では同梱 spec が平置きからパック形へ移ったのに、自分の repo には古い `specs/<subsidy_id>.json` が残ることがあります。`tools/update-core.sh` は manifest から消えたファイルを自動削除しないためです。この残留ファイルは、同じ `subsidy_id` の `specs/<subsidy_id>/<subsidy_id>.json` がある場合は無視されます。気になる場合だけ、公式募集要項と現在の作業状態を確認したうえで、古い平置きファイルを手動で削除してください。

自作パックを作業 repo（必ず private）にコミットして長期保存したい場合は、`.gitignore` の `input/` 除外をそのまま外すのではなく、コミットしたいパックだけを明示的に許可する方法を検討してください。会社名、見積、売上、講評などの機密情報が混ざっていないかを顧客本人が確認し、必要なら `input/spec/<subsidy_id>/` のみを例外にします。

## my-* コマンドの追加

標準コマンドでは足りない社内手順や自分用の確認観点がある場合は、`.claude/commands/my-<name>.md` という名前で command を追加します。

例:

```text
.claude/commands/my-local-check.md
.claude/commands/my-estimate-review.md
.claude/commands/my-supporter-handoff.md
```

`my-*` コマンドには、顧客本人の業務に合わせた質問、支援者へ渡すメモの形、社内承認前の確認リストなどを書けます。ただし、次の境界は守ってください。

- 公式募集要項、confirmed spec、`input/current-application.json` の内容を置き換えない。
- 補助率、補助上限、対象経費、締切、文字数などを推測しない。
- 申請代行、代理提出、本人に代わる完成判断を促さない。
- 出力先は原則 `input/` または `knowledge/` にし、コア層ファイルを不用意に変更しない。

`tools/update-core.sh` は `.claude/commands/my-*.md` を更新対象にしません。上流コアを取り込んでも、顧客本人が追加した `my-*` コマンドは残ります。

## knowledge/ の育て方

`knowledge/` は、申請を重ねるほど価値が出る育成層です。提出後または採否判明後に `/retrospect` を実行すると、次の場所に記録が増えます。

- `knowledge/records/`: `schemas/application-record.schema.json` に沿った JSON 記録
- `knowledge/lessons/`: 人間が読みやすい Markdown の学び

`/draft-section` と `/review` は、`knowledge/lessons/` に過去の学びがある場合、それを読んで次回の叩き台作成や定性レビューに反映します。たとえば、過去に「対象経費の根拠資料が遅れた」「課題と投資内容の接続が弱かった」「文字数制限を後で直す負担が大きかった」という lesson があれば、次回の draft や review で重点確認できます。

knowledge は、古い募集要項や別制度に依存している可能性があります。次回に使うときは、必ず今回の confirmed spec と公式募集要項に照らし、食い違う場合は今回の募集要項を優先してください。

## 上流更新を取り込む考え方

GitHub の `Use this template` で作った repo は、通常の fork ではありません。そのため、最初から上流テンプレート repo への `upstream` remote があるとは限りません。上流の改善を取り込むには、次のどちらかで上流ファイルを手元に用意します。

| 方法 | 向いている場合 | 概要 |
| --- | --- | --- |
| 上流 repo を別フォルダに clone する | Git 操作に慣れている | 最新のテンプレート repo を別フォルダに置き、自分の作業 repo からそのパスを指定する |
| release zip などを展開する | Git 操作を増やしたくない | 提供された release を別フォルダに展開し、そのパスを指定する |

上流 repo を remote として管理したい場合は、自分の作業 repo で `git remote -v` を確認し、必要に応じて `git remote add upstream 公式テンプレートrepoのURL` のように追加します。ただし、`tools/update-core.sh` は remote 名ではなく、手元にある上流 repo または展開済みフォルダのパスを受け取ります。

## update-core の手順

上流更新は、必ず dry-run で差分を確認してから apply します。

```bash
bash tools/update-core.sh --dry-run ../saita-kun-planner-upstream
```

dry-run では、`core-manifest.json` に載っているコア層ファイルだけを確認し、主に次の状態を表示します。

| 状態 | 意味 | 対応 |
| --- | --- | --- |
| `unchanged` | 自分の repo と上流が同じ | 何もしない |
| `new` | 上流に新しいコアファイルがある | 内容を確認して apply 候補にする |
| `changed` | 上流でコアファイルが更新されている | 自分の運用に影響がないか確認する |
| `user-modified` | 自分がコアファイルを編集しており、上流版で単純更新できない | 自動上書きせず、必要箇所だけ手で取り込む |

問題なければ apply します。

```bash
bash tools/update-core.sh --apply ../saita-kun-planner-upstream
```

特定の `user-modified` ファイルを上流版で上書きしたい場合だけ、内容を確認したうえで `--force-file` を使います。

```bash
bash tools/update-core.sh --apply ../saita-kun-planner-upstream --force-file docs/manual.md
```

apply 後は、必ず次を実行してください。

```bash
bash tools/validate.sh
```

検証が通らない場合は、上流更新と自分の変更が衝突している可能性があります。`input/`、`knowledge/`、`my-*` コマンドは更新対象外なので、まずは `core-manifest.json` に載っているファイルの差分を確認してください。

## result-report と update-core の関係

result-report は任意提出です。非提出でも、いま手元にある slash command、schema、spec、検証ツール、manual、template などの中核ハーネスは使えます。

将来、公式フィード接続、update-core の継続運用、最新 spec 受領などの公式サービスが用意される場合、それらは提出者への付加価値として連動することがあります。これは採否データ提出をキット利用の必須条件にするものではありません。B2B/有料契約では、result-report を出さずに同等機能を使える道を残します。

## 何をどこに書くか

| 書きたいもの | 置き場所 | 理由 |
| --- | --- | --- |
| 会社情報、見積、募集要項原文、draft | `input/` | 今回申請の作業情報であり、公開しない前提 |
| 募集要項から作った自分用 spec | `input/spec/` | bundled spec より今回の公式資料を優先するため |
| 提出後の結果、講評、次回の学び | `knowledge/` | 次回以降の `/draft-section` と `/review` が読める育成層 |
| 自分用の slash command | `.claude/commands/my-*.md` | コア更新から守るため |
| 標準コマンド、schema、tools、docs の共通改善 | 上流テンプレート側へ提案、またはコア層として慎重に編集 | 将来の `update-core` と衝突しやすいため |

迷ったら、顧客固有の情報は育成層へ、全利用者に効く改善はコア層へ分けてください。コア層を編集した場合は、次回の上流更新で `user-modified` になる可能性があります。
