# キット改善ループ（内部運用）

> **対象：提供側（内部運用）**。利用者向けではありません。
> 設計の正本は `docs/governance/data-charter.md`、データ説明は `TERMS.md` / `docs/data-policy.md`。
> 用途は**キット品質の改善のみ・外販しない**。

## ループ全体

```
任意 result-report + web 公示採否ラベル
   → allowlist 検査・秘匿処理
   → win-pattern 抽出
   → 勝ちパターン（採択要因の根拠つき）
   → キット改善（このリポの prompt / template / command / docs を更新）
   → Ralph で改善 story 化（.ralph/prd.json）→ validate 緑 → 反映
```

result-report は `docs/governance/data-charter.md` §4 の allowlist に限る任意 export です。申請本文、事業計画書本文、`input/` 配下の生データ、prompt、AI 応答、生ログ、Git remote、端末ユーザー名は使いません。

## 勝ちパターンをどこに反映するか

| 抽出された知見 | 反映先 |
|----------------|--------|
| 採択された叩き台に共通する構成・論点の並べ方 | `templates/事業計画書テンプレ.md` / `docs/事業計画書の構成.md` |
| 補助金種別ごとの要件対応の勝ち筋 | `templates/補助金要件マッピング.md` / `.claude/commands/subsidy-fit.md` |
| 採択につながった書き方の観点 | `.claude/commands/draft-section.md` |
| 不採択につながった弱点・指摘観点 | `.claude/commands/review.md` |

## 運用ルール

- 改善は **採択要因の根拠（judgment_basis 思想）** に基づくこと。「なんとなく良さそう」では反映しない（捏造防止）。
- 改善は Ralph の改善 story として起票し、`tools/validate.sh` の既存アサーションを弱めないこと。
- result-report は任意提出です。非提出でも中核ハーネスは使えるため、改善ループ側は提出済みデータだけを扱います。公式フィード接続や update-core などは、提出者への付加価値に連動することがあります。
- B2B/有料契約の顧客は、データ提供なしで同等機能を利用できる契約上の逃げ道を持ちます。
- 改善の元データは allowlist 内の構造化メタデータと採否結果に限定し、生機密（`input/`）は使いません。
- 個別の result-report は第三者・スポンサー・B2B 顧客に渡しません。公開する場合は秘匿処理済みの集計だけにします。

## 注意

旧 private repo 観測・harvest 前提の運用は撤回済みです。現行ループは、任意 result-report と web 公示などの公開情報を、データ憲章の非接触原則に沿って扱います。result-report の送信実装が入るまでは、`/retrospect` で作られる `knowledge/records/` の application-record をユーザー自身の repo 内に残すところまでがこのキットの範囲です。
