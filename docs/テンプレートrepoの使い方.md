# GitHub テンプレート repo として使う

このページは、Claude Code 契約者が saita-kun-planner を自分の作業用リポジトリとして用意し、補助金申請用の事業計画書の叩き台づくりを始めるための短い手順です。

この repo は顧客本人が作成主体として使うスターターキットです。作成者は顧客本人です。AI は補助、壁打ち、整理役であり、申請代行、代理提出、本人に代わる完成判断はしません。行政書士法に抵触する使い方を避けるため、最終的な内容確認、修正、提出判断は必ず顧客本人が行ってください。

## 1. Use this template で自分の repo を作る

GitHub でこのテンプレート repo を開き、`Use this template` を選んで自分の作業用 repo を作ります。補助金申請では会社情報、売上、従業員数、投資額、見積額、取引先情報、未公開の事業計画を扱うため、作業用 repo は必ず private にしてください。

公開 repo にした場合、`input/` は `.gitignore` で除外されていても、誤って別ファイルに実データを書き込むリスクがあります。共有用のサンプルを作るときは、会社名、金額、個人情報、固有の取引先名を含めず、架空の値であることを明記してください。

## 2. clone して Claude Code で開く

作業端末で自分の repo を `git clone` し、clone したフォルダをルートとして、新しい Claude Code セッションを開きます。このキットは Claude Code の slash command を前提にしているため、リポジトリのルートフォルダで作業してください。

最初に `/setup` を実行し、環境セルフチェックと利用規約の同意確認を済ませます。`/setup` が green になったら `/start` を実行すると、前提、作業順、必要な入力、法務ガードレールを確認できます。迷わなければ、同梱 spec を使う場合は `/select-subsidy`、自分の募集要項から spec を作る場合は `/ingest-guidelines` → `/confirm-spec` で confirmed spec を先に確定します。必要に応じて `/build-pack` で書き方メモを作り、その後 `/intake`、`/subsidy-fit`、`/plan-deliverables`、`/draft-section`、`/review`、`/verify`、`/finalize` の順に進めます。

## 3. 自分の `input/` を用意する

`input/` には、顧客本人の実情報を置きます。たとえば、会社概要、沿革、従業員数、売上規模、現状課題、投資計画、見積書メモ、補助金の公式募集要項メモ、作成中の事業計画書の叩き台などです。

`input/` は gitignore 済みです。ただし、gitignore はすべての情報漏えいを防ぐ仕組みではありません。実データを `docs/`、`templates/`、`examples/`、README、issue などに転記しないでください。Claude Code に貼る情報も、顧客本人が開示してよい範囲を判断してください。

## 4. 公式の募集要項を一次情報にする

補助率、補助上限、対象経費、対象者、提出期限、文字数、添付資料、審査項目、様式指定は、必ず公式の募集要項で確認してください。この repo の説明や AI の出力と募集要項が食い違う場合は、募集要項を優先します。

数値は推測しません。顧客資料または募集要項から確認できない売上、効果見込み、KPI、投資額、補助金額、採択率などは断定せず、`[要確認]` を付けて顧客本人が確認します。

## 5. テンプレート本体と作業用 repo を分けて考える

この repo は、補助金申請者が自分の作業環境を作るためのひな形です。テンプレート本体に顧客の実データを入れる必要はありません。顧客ごとの作業は、`Use this template` で作った自分の repo の中で行います。

テンプレート本体への質問や改善提案を GitHub issue で送る場合も、実データは貼らないでください。再現に必要な説明は、架空の会社、架空の補助金名、架空の数値に置き換えてください。
