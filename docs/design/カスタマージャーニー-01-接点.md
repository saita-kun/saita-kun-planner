# カスタマージャーニー §1 — 接点フェーズ 要件定義

> status: 確定（2026-06-16 ヒアリング3ラウンドで分岐解消）
> scope: **事業者直販 × X 入口** の接点フェーズのみ。キット以降・採否・データ回収運用・支援者卸は別フェーズ。

---

## 0. このフェーズの定義

| | |
|---|---|
| 入口 | X 発信での認知 |
| 出口 | 診断結果を受け取った人の「申請したい」転換点（＝キットフェーズへの handoff） |
| 主体 | 事業者直販（支援者卸は別ジャーニー） |
| 前提 | **接点では Claude Code 不問**。母集団を広く保つ |

接点フェーズの設計思想: **水口は広く、足切りは奥で**。X→診断までは誰でも入れ、Claude Code 必須の足切りは「申請したい＝キット」段階で初めて課す。

---

## 1. 対象（who）

- 「補助金を申請できるか分からないが**興味はある**」**小規模事業者・個人**。300万円未満補助金の対象層
- 既存実績16社（ものづくり補助金・開発会社経由のリテラシー高め層）とは**別の、より手前の層**を狙う
- このフェーズのジョブ: 「自分に合う補助金があるのか、まず知りたい」

---

## 2. 認知チャネル

- 主チャネル: **X 発信（既存運用）**
- トーン: 強い批判表現は**前面に出さない**。「自分で補助金を探せる／取れそう」のマイルドな訴求
- 動線: X 投稿 → LP 着地

## 3. ランディングページ（LP）要件

### 3.1 配置・置換
- HP 内に新設し、**既存「サービス（代行）」ページを置換**する
- HP 全体の**思想が強い部分をマイルド化**する
- 対象サイト: `https://www.subsidy-support.tech/`。公開サイト側で管理し、この template repo には接点設計だけを残す

### 3.2 メッセージ方針
- **匿名**（個人・思想を前面に出さない）
- 「**自分で申請できそう**」と思わせる
- 接点段階では result-report を送信しない。キット段階で result-report を扱う場合は、任意提出・allowlist・非提出でも中核機能が使えることを明示する

### 3.3 行政書士法 表現制約（必須順守）
正本: [`docs/法務とスコープ.md`](../法務とスコープ.md) と [`docs/governance/data-charter.md`](../governance/data-charter.md)
- 構図は常に「**顧客本人が作成、AI はアシスト**」
- 課金は「書類作成報酬」ではなく「**ツール利用料**」フレーム
- 「サイタくんが申請書を作成」と取れる表現は**禁止**

### 3.4 CTA
- 主 CTA: **キット（GitHub 公開テンプレート `saita-kun-planner`）へ誘導**
- 副 CTA: **補助金検索（`/subsidy-search`）**・LINE 購読（`/subscribe`）
- ＜2026-07-06 更新＞ 当初案の「診断フォーム（Tally/Typeform）へ誘導」は、公開診断フォームが未実装のため変更。診断相当の無料体験は当面 `/subsidy-search`（無料の補助金検索）が担い、専用診断フォームの新設は §4 のフロー実装時に再検討する

## 4. 接点の中核フロー（intake → 決済 → 配信）

```
X 発信
  │
  ▼
LP 着地（既存サービスページを置換）
  │  CTA: 診断を受ける
  ▼
フォームでプロファイル入力（無料・必要最小限）
  │
  ▼
マッチング診断 実行（3段階・説明可能性付き）   ← 初回無料
  │
  ▼
結果配信：LINE Push ＋ メール等
  │
  ▼
「申請したい」転換点  →  申請支援（キット）フェーズへ handoff
                         ここで Stripe 決済（初回から有料）＋ Claude Code 必須 を同時に課す
```

接点（マッチング診断）は**無料で価値を体験**させ、課金とClaude Code足切りは**申請支援（キット）の handoff に集約**する。決済ゲートと足切りが同じ段でそろう。

### 4.1 intake
- 入力項目: 基本情報、事業概要、規模、地域、希望カテゴリなど、接点診断に必要な最小限の情報
- 受信: 接点側の別システムで JSON 化し、キット本体の `input/` 生データとは混ぜない

### 4.2 診断
- エンジン: 3段階の explainable matching。根拠は募集要項・公開情報に結び、採択可能性を断定しない
- 出力: 診断結果メモ。judgment_basis は根拠条文や出典に結び、出典不明は `[要確認]` とする

### 4.3 配信
- LINE やメールなど、利用者が明示的に登録したチャネルで配信する
- 配信停止や購読状態は接点側で管理し、キット本体の `input/` とは分離する

## 5. 課金（Stripe・新規構築）

| 対象 | 課金形態 | タイミング | 集金 |
|------|----------|-----------|------|
| マッチング診断 | 従量（診断1回単位） | **初回無料**（水口を広く取る。2回目以降の従量課金は余地として残す） | Stripe（初回は不要） |
| 申請支援（キット＝ハーネス＋Agent サンドボックス/GitHub） | スポット | **初回から有料** | Stripe |

> ＜2026-07-06 注記＞ 本表のキット課金（スポット・初回から有料）は 2026-07-02 の DRI 決定で撤回。現行は事業者直販＝キット無料/安価（収益は法人スポンサー・支援者/B2B 層）。正本: `docs/licensing-tiers.md`（発効 2026-07-05）・`TERMS.md` 第4条。本表は旧設計の記録として残す。

- 課金は**申請支援（キット）に集約**。接点（マッチング診断）は初回無料で価値体験に徹し、**決済ゲートと Claude Code 足切りを申請支援の handoff に同時に置く**
- result-report は任意提出です。非提出でも中核ハーネスは使えます。提出者には公式フィード接続、update-core、最新 spec 受領などの付加価値を紐づけることがありますが、採否データ提供をキット利用条件にしません

## 7. フラグ（要 DRI ／ 後続対応・本書の範囲外だが明記）

1. **result-report 任意性の維持**: 採否データ提供を利用条件にしない。非提出でも中核ハーネスは使え、提出者には公式フィード接続や update-core などの付加価値を紐づける方針にそろえる
2. **法務ゲート（DRI 直管理）**: LP 文言・課金表現（ツール利用料）・result-report 任意性・規約整合は DRI 法務 pass 対象。AI 側で法解釈はしない
3. **公開サイト側の実装管理**: LP 置換・マイルド化の具体作業は公開サイト側の管理範囲で進め、この template repo には利用者向けキットの構成だけを置く
4. **マッチング 2回目以降の課金（要確定）**: 接点は初回無料で確定。2回目以降を従量課金にするか完全無料にするかは接点 KPI（LP→診断→申請支援 CVR）計測後に決める。＜旧懸念「初回課金のファネル摩擦」は、課金を申請支援へ寄せることで解消＞
