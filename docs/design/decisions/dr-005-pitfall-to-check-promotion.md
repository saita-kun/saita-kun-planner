# DR-005: 落とし穴は文書でなく検査・観点へ昇格させる

> 種別: 設計不変条件 ／ 状態: 有効 ／ 決定日: 2026-07-19 ／ 決定者: DRI ／ 出典: 内部決定記録（非公開）

## 決定

制度・書き方の「落とし穴」の知見は、読み物として蓄積せず、**コア層の検査・観点へ昇格**させる。

- 制度事実の穴 → spec の clause / confirmation へ
- 書き方・審査の穴 → `/review` の重点観点・禁止表現検査・coverage gap 検査へ

## 理由

- AI 中心運営の前提では、「読んで気をつける」知識は実行時に効かない。**落とし穴は AI に読ませるのではなく実行させる**（機械が検出する形にする）方が確実。
- result-report → キット品質改善の流れ（`TERMS.md` §3）の具体化にあたる。改善の器は既存の改善ループであり、新設の読み物ではない。

## 制約（運用 AI が守ること）

- 落とし穴を発見したら、「落とし穴集.md」のような新規文書を作って終わりにしない。spec の clause/confirmation、checker、`/review` 観点、validate アサーションのいずれに昇格するかを判断して起票・実装する。
- `knowledge/`（個社 private 育成層）の位置づけは変更しない — そこは顧客自身の学びの層で、コア層への昇格とは別物。
- 昇格した検査を「注意書きに戻す」逆行（検査の削除・注意文への置換）をしない。

## 違反例

- レビューで見つけた頻出ミスを `docs/よくある失敗.md` に追記して完了とする。
- checker の FAIL を「手引きに書いてあるから」と WARN に格下げする。
