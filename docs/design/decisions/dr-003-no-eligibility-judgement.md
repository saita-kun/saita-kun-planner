# DR-003: 適格性を「判定しない」— 叩き台生成に限定

> 種別: 設計不変条件 ／ 状態: 有効 ／ 決定日: 2026-07-03（記録化 2026-07-24） ／ 決定者: DRI ／ 出典: 内部決定記録（非公開）

## 決定

本キットは適格性・採否の「判定者」にならない。役割は、公式の募集要領との**出典付きの突合・整理・叩き台生成**に限定する。作成者は顧客本人であり、完成判断と提出は必ず人間が行う。

## 理由

- 行政書士法の境界（申請代行・代理提出・代書の禁止）。本キットは利用者自身が自分の AI で自分の書類を作る自己完結型ソフトであり、提供側は書類作成に一切関与しない（非接触原則。正は `docs/governance/data-charter.md`）。
- 「LLM は判定根拠を検証可能に説明できず、誤判定に不服を申し立てられない」という批判に対し、「判定しない」ポジション自体が防御になる。出典付き突合・下書きに限定する設計は公共 AI の国際的な収斂点と同型。

## 制約（運用 AI が守ること）

- `/subsidy-fit` などの照合結果は「要件と現状の突合結果＋不足の指摘」であり、「対象です／採択されます」という断定にしない。
- 成果保証・全自動・代行を意味する表現を顧客向けテキストに書かない（DR-006 の禁止表現検査の対象）。
- 数値は推測しない。出典不明の事実には `[要確認]` を付ける。公式の募集要項と本リポジトリの記述が食い違う場合は募集要項を優先する。
- 提出工程を自動化しない（完全自動化しない原則）。

## 違反例

- 適格性や採否の見込みを断定する応答（「対象です」型・結果を請け合う型）を実装する。
- 申請書が人手なしで勝手に完成・提出まで進むかのような導線文を README に書く。
- 効率化のため `/finalize` から提出 API 連携を生やす。
