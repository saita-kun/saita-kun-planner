# DR-004: 上流非依存で自走可能（継続性の設計不変条件）

> 種別: 設計不変条件 ／ 状態: 有効 ／ 決定日: 2026-07-19 ／ 決定者: DRI ／ 出典: 内部決定記録（非公開）

## 決定

上流（canonical repo）が停止しても、各 fork の AI が**公式公募要領から入口B（`/ingest-guidelines` → `/confirm-spec`）で spec を自作し、自走継続できる**構造を維持する。継続性の保険は人の引き継ぎ段取りではなく、この構造的独立性で担保する。

## 理由

- 中央のデータ更新者・中央サーバーに依存した公共財は、担い手が抜けると静かに死ぬ（先行事例で実証済みの死因）。本キットは「利用者自身の AI で動く＝中央サーバー不要」でこれを構造的に回避している。
- spec レジストリ・同梱パックは**利便レイヤ**であり生命線ではない。生命線は入口Bの自作経路。
- 運営者交代は「別の AI インスタンスに同じ repo を渡す」だけで成立するよう、運営作業（export・update-core 配布・公開ゲート）を AI が実行可能な手順として repo に残す。

## 制約（運用 AI が守ること）

- 入口Bを弱める変更をしない: 同梱パック前提の必須化、レジストリ参照の必須化、中央 API・外部サービスへの実行時依存の導入はすべて設計違反。
- 中核機能（叩き台生成まで）は、ネットワーク上の上流資産が消えても手元の repo と公式公募要領だけで完結すること。
- 配布アーキテクチャ（repo=育成単位／パック=配布単位／完成環境=生成物）の正は `docs/design/repo-structure.md`。矛盾する変更はそちらの改訂とセットでのみ行う。

## 違反例

- `/select-subsidy` を「レジストリから取得」必須にし、入口Bを非推奨に格下げする。
- spec の検証に中央サーバーの API 呼び出しを組み込む。
- 「メンテナ不在時は利用を停止してください」という運用前提を文書に書く。
