# 設計: リポジトリ構造と配布アーキテクチャ（repo / パック / 完成環境の分離）

> 本ファイルは saita-kun-planner の「何を 1 リポジトリで育て、何を補助金ごとに配るか」の設計正本。
> 決定日: 2026-07-17（DRI: 代表）。`docs/design/harness-pack-builder.md`（3層ハーネス構造）の補完であり、同設計の L1/L2/L3 の語彙を前提とする。
> 公開版では、利用者・貢献者が把握すべき設計原則と方向だけを記録する。

## 0. 決定（原則）

**repo は育成単位、補助金パックは配布単位、完成環境は生成物。**

1. **L1 共通ハーネスは本リポジトリ 1 つを正史に固定する**。slash command・状態機械・検証器・スキーマ・builder（パック生成機構）は、ここでだけ育てる。
2. **補助金ごとの差分はすべて L2 パック（データのみ）が運ぶ**。`harness-pack-builder.md` §0-2「L2 にコードを置かない」の帰結として、補助金ごとに実行系を複製する理由は存在しない。
3. **補助金ごとのリポジトリ分割はしない**（検討のうえ却下。§1）。「特定の補助金の完成環境をすぐ使いたい」というニーズは、リポジトリ分割ではなく配布チャネル（§3）で満たす。

## 1. 採らなかった案: 補助金ごとにリポジトリを分割する

1 repo = 1 補助金とし、その中にハーネスとパックを同梱して配る案。「その補助金用の完成環境を clone 一発で入手できる」利点があるが、次の理由で却下した。

- **ハーネス改善の伝播が破綻する**。法務ガードレール文言・スキーマ・検証器・`update-core` の修正を全リポジトリへ配り直す作業が発生し、対象補助金が増えるほど保守が線形に重くなる。全補助金（自治体ロングテール含む）を扱う方針と両立しない。
- **コードの版とデータの版が混ざる**。補助金は公募回ごとに要領が改訂される。リポジトリ＝補助金にすると、ハーネスの更新履歴とパックの更新履歴が 1 つの履歴に混在し、どちらの周期でも追えなくなる。
- **横断育成が構造的に阻害される**。Issue・PR・利用者の学びがリポジトリごとに分散し、ある補助金で見つかった検証器の穴が他の補助金の利用者に届かない。

なお「ハーネスとパックを一緒に配る」こと自体は、分割しなくても既に成立している。本リポジトリは builder を L1 に内蔵し（`harness-pack-builder.md` §0）、同梱パックを `specs/` に持つ。分割案が本当に足そうとしていたのは「同梱されていない補助金の即時入手」であり、それは §3 の配布形で解く。

## 2. 横断育成ループ（builder が 1 系統であることの帰結）

提供側も利用者も同じ builder（`/ingest-guidelines` → `/confirm-spec` → `/build-pack`）を使う（`harness-pack-builder.md` §9）。このため、新しい補助金を 1 件パック化するたびに、ハーネス改善の材料が単一リポジトリへ還流する。

- `predicate_state = not_encodable` の記録 → スキーマの表現力ギャップの検出信号（同 §6.2）
- `check-spec` / `check-pack` の FAIL パターン → 検証器・抽出規律の改善候補
- 抽出スポットチェックの不一致（同 §7.2） → 原本取込フローの改善候補

これらは GitHub の Issue / PR として本リポジトリに集約され、修正は `update-core` の配布対象となって適用した利用者環境へ、CI（`tools/validate.sh`）の再検証で全同梱パックへ波及する。リポジトリを分割しないことが、この一点集約の前提条件である。

## 3. スケール時の配布形（方向の固定）

対象補助金が増えたときの配布は、次の 3 段で構成する。方向のみ本書で固定し、実装は従来どおり次フェーズ（§4）。

| 段 | 形 | 役割 |
| --- | --- | --- |
| 同梱パック | 本リポジトリの `specs/<subsidy_id>/` | 代表・高需要の少数に絞る。全補助金の同梱はしない（clone と検証対象の肥大を避ける） |
| パックレジストリ | 公開 data リポジトリ（パック単位のディレクトリ + 索引 + sha256 台帳） | 補助金ごとの配布単位。`pack.json` は sha256 台帳付きでそのまま配布単位になる設計済み（`harness-pack-builder.md` §3）。L2 のコード禁止原則はレジストリ側でも維持する |
| 生成 starter | 本リポジトリ + パックから機械生成する配布物 | 「補助金別の完成環境」が必要な場合（発見性・導線）の受け皿。read-only の生成物とし、一次保守単位にしない。人手で直接育てるリポジトリを増やさない |

- 利用者リポジトリ内の解決優先順（`harness-pack-builder.md` §2: `input/spec/` ＞ 同梱パック）は不変。レジストリから取得したパックは resolver 上は同梱パック相当として扱う。物理配置と `core-manifest.json` 完全性検査との整合（`specs/` 配下は現在、全ファイルが manifest 登録対象）はレジストリ実装フェーズで決める。
- レジストリからの取得口（パック導入コマンド）は本リポジトリ側に持たせる。取得したパックの完全性は `pack.json` の sha256 台帳と `check-pack` で検証する。

## 4. やらないこと（本設計のスコープ外）

- パックレジストリ・パック導入コマンド・starter 生成の実装（本書は方向の固定のみ。`harness-pack-builder.md` §12 の据え置きを維持する）
- 既存構造の変更（`specs/` 配下・`core-manifest.json` 運用・resolver 規約は現行のまま）
- 同梱パックの選定基準の策定（レジストリ実装フェーズで扱う）
