## 0. 前提 — サービスの根幹（見失わないための固定）

- **提供物は2つだけ**: (1) クライアントが持ち込む AI が使う**ハーネス**（slash commands + スキーマ + 検証ゲート + マニュアル）(2) **初回プロンプト**。これ以外は提供しない。
- **出力される成果物**: 人がやることリスト・マニュアル（GitHub アカウント作成手順等）+ ユーザー自身が作る事業計画書・タスクリスト。
- **体験**: ユーザーが出したい補助金の募集要項を登録 → AI+ハーネスが質問 → ユーザーが回答 → 事業計画書とタスクリストが生成・更新されていく。
- **非接触原則（本憲章の第一原則）**: 事業計画書・申請本文・`input/` の生データは**提供側が構造上一切触れない**。処理は全てクライアント側で走り、成果物はユーザー自身が作る。提供側に還流するデータは §4 の result-report（採否結果+構造化メタデータのみ）に限られ、これは「後から絞った」ものではなく**設計上そもそもアクセスできない範囲を明示した狭い任意 export**である。
- この非接触は行政書士法スコープ（`docs/法務とスコープ.md` 正本）の技術的裏づけであり、データ設計の全ての章はこの原則に従属する。

## 1. 目的と原則

本憲章の目的は、OSS として最大リーチを取りつつ、中核資産（事業者プロファイル×補助金適格条件×採否結果の3点セット）と「サイタくん」ブランドの信頼を長期に守ることにある。

**レイヤー分離（設計の根幹・混同禁止）**: 「本家=正史」と「売却不能」はコードのライセンスでは守れない。守る対象を層で分けて設計する。

| レイヤー | 守る手段 | 本憲章の章 |
|---|---|---|
| A: 配布の許諾 | OSS ライセンス（Apache-2.0） | §3 |
| B: 貢献の権利集約 | CLA なし / DCO | §3 |
| C: 名乗りの独占（ブランド・正史） | 商標登録 + TRADEMARK.md | §2 |
| D: 会社そのものの売却可否 | 定款・会社形態 | §9 注記（DRI 領域） |

- OSI 定義上「OSS なのに商用・販売だけ禁止」はできない。したがって守る対象はコードではなく**商標・公式フィード・result-report データ・正史運用**に置く。
- 原則: (1) 非接触（§0）(2) データ最小化 (3) 個票は内部・公開は秘匿集計のみ (4) スポンサー資金は診断・推薦に一切影響しない (5) 透明性（収集項目・ロジック・資金使途を公開）。

## 3. OSS ライセンスと貢献ルール

- **配布ライセンス = Apache-2.0**。
  - 理由: permissive で「template を複製して自分のものとして使う」利用形態・クライアント側実行と整合。§6 で「コードの許諾と商標は別（このライセンスは商標使用権を与えない）」を明文化でき、特許グラント・NOTICE を備え B2B（士業白ラベル・金融機関組込）に説明しやすい。
  - 不採用: **AGPL**（クライアント側実行・自社ホストなしのためネットワーク条項が発火せず、コピーレフト伝播の摩擦だけが残り「自由に複製してほしい」思想と逆行）。**CC**（CC 自身がソフトウェアに非推奨、非互換）。
  - repo 全体を単一ライセンスで覆う（マニュアル散文だけ CC-BY 等の分割はテンプレ配布では過剰）。
  - 注: MIT でも大枠は成立する（実害差は小）。Apache-2.0 を採る実益は §6 明文化・特許グラント・体裁。ブランド保護の実体は §2 の商標であってライセンス条項ではない。
- **貢献管理 = CLA なし / DCO 必須**。
  - CLA を置くと著作権が機構(株)へ集約され、将来の独占再ライセンス・売却疑念（rug-pull）を招く（MongoDB/Elastic/Redis 型）。CLA を置かず著作権を貢献者に分散させることで、**全貢献者の合意なしに独断で proprietary 化できない**耐性と fork 保険を得る（Godot/Blender 前例、Software Freedom Conservancy「CLA は要らない」）。
  - 運用: DCO v1.1 を必須化。各コミットに `Signed-off-by`（`git commit -s`）、`dcoapp/app` を org に入れて required status check にする。sign-off 情報が公開履歴に残る旨を CONTRIBUTING に明記。
- **重要な限界（レイヤー D、§1 の表参照）**: DCO/CLA なしが担保するのは、貢献者の権利保全と fork 可能性である。会社形態・商標・公式資産の支配に関する制約はコードライセンスではなく、§2 の IP 帰属、§7 の独立性、§9 の非営利 data steward 移行条件で別レイヤーとして扱う。
  - レイヤー D をさらに構造化する会社形態（保証有限責任会社／steward-ownership 等）は、税務・事業・不可逆を伴う DRI（代表）の意思決定であり、詳細は内部決定記録（非公開）で管理する。本憲章では、公開利用者に影響する原則と移行条件だけを示す。

## 4. result-report の収集範囲（allowlist / denylist）

提供側に還流するデータは result-report のみ。収集は**明示的 allowlist に列挙した項目に限る**（列挙外は集めない）。

**収集する（allowlist）**:
- メタ: `report_id`, `schema_version`, `core_version`, `submitted_at_month`
- 補助金: `subsidy_id`, `subsidy_name`, `round`, `jurisdiction_level`
- 結果: `result_status`（採択/不採択/取下げ等の enum）, `result_month`, `requested_amount_band`（レンジ）, `awarded_amount_band`（レンジ）
- 事業者プロファイル（すべて band/enum、生値でなく区分）: `industry_major/minor`, `prefecture_or_region`, `employee_count_band`, `revenue_band`, `years_in_business_band`, `corporate_type`
- 適格条件: eligibility の boolean/enum フラグ（§3 スキーマの predicate 評価結果）
- 検証情報: validate pass/fail, missing field 数, schema errors

**収集しない（denylist、明示）**:
- 申請本文・事業計画書本文、会社名・代表者名・住所詳細・電話・メール・URL、法人番号、口座/税務情報、添付資料、prompt / AI 応答の生テキスト、生ログ、Git remote、端末ユーザー名。**自由記述欄は原則禁止**。
- `input/` 配下（生機密）は `.gitignore` で隔離され、そもそも提供側に到達しない（§0 非接触）。

## 5. 任意性・consent・削除・保管

- **任意性（同意の実質を守る設計）**: result-report の提出は**任意**。**非提出でも中核ハーネス（ローカル実行の全機能）は使える**。提出者に紐づくのは「公式フィード接続・update-core・最新 spec 受領」という**継続的な公式サービス**であり、これは非提出者を機能排除するものではなく提出者への付加価値として設計する（2026-07-02 DRI 決定「特典と報告実績を紐付け」を、同意の自発性を保つ形で具体化）。
- **交換の透明化と逃げ道（GitLab Registration Features 型）**: 「result-report 提出 = 無料利用者が公式サービス特典を得る対価」という交換を規約で明示し、**B2B/有料契約の顧客はデータ提供なしでも同等機能を利用できる**逃げ道を残す（同意の任意性を担保する構造。GitLab は同型で「有料 tier からは機能を取り上げない」設計）。
- **consent 取得**: 提出コマンドは送信前に「何を送るか」の JSON プレビューを表示し、`--consent result-report-vYYYY-MM-DD` または対話確認で **consent version を記録**する。技術強制はしない（OSS）。採否＝価値ある個票のため、匿名テレメトリの opt-out とは区別し**明示的 opt-in** とする（Angular CLI/Nuxt 型）。
- **FTUX バナー + ドライラン**: 初回実行時に「何を・誰が集め・どう無効化し・詳細 URL」を4文で示す告知バナーを出し、**告知前は一切送信しない**（Homebrew 型）。`RESULT_REPORT_DEBUG=1` 相当で**送信せず内容を stderr 表示するドライラン**を必ず提供する（Homebrew/Next.js 型）。
- **士業・支援者による代理提出**: 顧客本人/法人を代表して consent する権限を代理者が取得済みである旨の**表明**を要求する（守秘義務・利益相反の配慮。専門家を含む運用条件の詳細は内部決定記録（非公開）で管理する）。
- **削除・撤回**: 提出時に削除用 ID / receipt を発行。raw store から該当レコードと紐付け情報をパージする。**既に公開・集計済みのスナップショットは遡及変更しない**が、次回再集計から除外する（完全匿名化された集計は個人データではないという実務に沿う）。
- **開示ドキュメント**: 収集項目・目的・保存先・保持期間・オプトアウト/削除手順を専用ページで全公開する（Homebrew / Next.js / CNCF telemetry ガイドライン水準）。

## 6. 匿名集計と公開基準

- **個票と集計の二層分離（Five Safes 運用）**: 個票（microdata）は**内部限定/enclave 隔離**（「政府があなたに出せる補助金はこれ」予測モデルの学習等）。**対外公開は秘匿処理済みの集計のみ**。row-level 公開は禁止。将来の研究提供も「ダウンロード」でなく enclave 出力搬出方式とし、ONS の Five Safes（Safe projects/people/settings/data/outputs）を運用フレームとする。
- **集計データの外部配布ライセンス**: result-report 由来の秘匿集計を外部に再利用可能な形で配布する場合は **CDLA-Permissive-2.0**（Linux Foundation のデータセット用ライセンス）を受け皿とする。
- **匿名性の閾値（証拠ベースの段階値）**:
  - 件数のみの度数表 → **k ≥ 5**。**採否（機微値）を含む集計 → k ≥ 11**（CMS「セル値11未満は非表示」・EMA/Health Canada の臨床データ公開水準に合わせる。件数だけなら公的統計標準の k=3 でも成立するが、本件は準識別子合成の再識別リスクが高く採否が機微なため厳しめに取る）。初期運用ではさらに保守的な公開 k≥20 も検討し、実測で調整する。
  - **small cell suppression**: 一次抑制 = 事業者数 **n<5 非公開**（採否を含む表は **n<11**）。二次（complementary）抑制で行・列・総計からの逆算を防ぐ。**表間秘匿を必須**とし、任意軸クロスのオンデマンド集計生成はしない（composition 攻撃の防止）。丸めは random/controlled rounding（conventional rounding は差分攻撃に弱く不採用）。抑制パラメータ（n, k, p）は**非公開**。
  - **l-多様性を必須**: センシティブ属性である**採否結果があるセル内で全員同じ（全員採択/全員不採択）だと個社の結果が判明する**（同質性攻撃）。同値クラス内で採否に複数値があることを要求し、偏るセルは統合・秘匿する。
  - 採択率の分子・分母とも**最低5件以上**。
- **準識別子の粒度管理**（「地域×業種×補助金名×時期」の合成一意化を防ぐ）:
  - 時期 → 年度/四半期に丸める（月日を出さない）。
  - 地域 → 都道府県 or 地方ブロック（市区町村は出さない）。稀少業種×小規模県は広域化。
  - 業種 → 日本標準産業分類の中分類/大分類（細分類は内部限定）。
  - 補助金 → 全国区大型はセルが厚く安全。**自治体独自の小規模補助金は申請者数が少なく即特定されうるため、地域別内訳は非公開**か種別カテゴリに丸める。
- **第三者提供**: スポンサー・B2B へ**個票は渡さない**（提供は秘匿集計のみ）。データ利用者に**再識別行為を契約で禁止**する。

## 7. スポンサー独立性と COI

- **中核不可侵原則（憲章冒頭級の宣言）**: **「補助金の推薦順位・診断結果は売り物ではない（The recommendation is not for sale）」**。Godot が「Godot can't be sold」を不可侵原則に掲げたのと同型に、これを製品 UI・規約・スポンサー契約すべての先頭に掲げる。
- **技術的中立の宣言（ASF 水準で明文化）**: 「スポンサーシップは**推薦順位・診断結果・補助金フィード採用・重み付け・検証ゲートに一切影響しない**」。スポンサー特典はロゴ掲載・謝辞・ケーススタディ・イベント枠までに限定する。得られるのは**可視性であって影響力ではない**。
- **ファイアウォール（実装レベル）**:
  - ranker / 診断パイプラインの入力に `sponsor_id` を渡さない。
  - スポンサー掲載フィードと補助金 eligibility フィードを**別 namespace** にする。
  - 診断ロジック・重み付けの変更は **public PR + test + changelog + COI 開示**を必須にする。ロジックを OSS として検証可能にし、pay-to-win でないことをコードで証明する。
  - スポンサー関連事業者の補助金情報を追加する場合も、同一の公開ソース基準・レビュー基準を適用する。
- **UI 分離**: スポンサー表示は「PR/スポンサー」と明示ラベルを付け、推薦・診断結果とは別 UI 領域に置く（Wirecutter 型の編集独立）。
- **COI 開示・除外**: 意思決定者は**半年ごとに COI 開示フォームを提出**し、利害ある議決から除外（recuse）する。関連営利の役員兼任・資本関係がある場合、可決には**利害関係のない者の過半数**を要する（Godot Section 13 / PSF 型）。禁止行為を列挙: スポンサーによる順位・スコア・診断結果の要求／購入／事前閲覧による改変。
- **開示（Godot × W4 の教訓）**: スポンサー一覧・金額規模・関連営利・資金使途を定期開示（可能なら監査付き）。Godot は財団理事が営利 W4 Games の経営を兼務したことが利益相反として炎上し、(1) COI ポリシー成文化 (2) 監査付き財務の定期公開 (3) 技術統治の独立明文化 (4) **関係企業も「他社と同条件」と宣言**（W4 が owner にならない）で応答した。**兼任・関連営利は避けられなくても、これらを炎上前に用意すれば疑義は大きく減る**。本件でも機構(株)自身や関連法人スポンサーを「推薦順位では他と同条件」と先に宣言する。
- **契約条項**: スポンサー/B2B 白ラベル契約に「順位・診断への非影響」「個票非提供」「再識別禁止」を明記し、「当社に有利な診断」の要求を先に潰す。**ロジック変更の監査証跡**を残し「スポンサー起因の順位操作はない」を後から検証可能にする。

## 8. 公式フィード / 診断ロジックの変更手続き

- 生 spec フィード（レジストリ）は無料公開し、OSS 利用者が検証できる形で維持する。有償機能や連携機能を設ける場合も、スポンサーや契約先が推薦・診断に影響しないよう §7 の独立性原則を優先する。
- 「公式フィード接続」が pay-to-play に見えないよう、接続条件（report 提出との紐付け）と、フィード内容がスポンサーに左右されないこと（§7）を明示する。
- 診断ロジックの変更は §7 の手続き（public PR + test + changelog + COI 開示）を通す。

## 9. 非営利 data steward への移行条件

将来「採否データが公共性を持った段階で非営利 steward（一般社団法人等）へ移す」二段階設計。**発動条件を規約に先置きする**（過去 consent に steward 移転を含めておかないと移行時に再同意が要る、という落とし穴を回避）。

- **発動条件（いずれか）**:
  - 有効 result-report が累計 **2,000件以上**
  - distinct submitter が **500以上**
  - 対象補助金 round が **50以上**
  - 都道府県/広域地域の coverage が **20以上**
  - 外部利用者（士業・金融機関・研究者・自治体等）が **3カテゴリ以上**
  - 公開集計/API が第三者の意思決定に使われ始めた
  - 機構(株)の支配権変更・事業売却・破産・データ独占ライセンス提案・単一スポンサー依存が年間収入の **40%超**
- **発動後**: 90日以内に移行計画、180日以内に一般社団法人へ result-report の raw/派生データ・consent 記録・スキーマ・集計パイプライン・公開 API の stewardship を移す。機構(株)は商用サービスを継続できるが、**データ利用条件は他社と同一**にする（自社優遇しない）。
- **落とし穴**: 早すぎる移行は運用が重い、遅すぎると「公共データを私企業が握る」批判。一般社団は法定費用約11万円・社員2名以上で登記のみのため、条件充足時に機動的に設立できる。
- **レイヤー D との接続**: 非営利移行後の steward-ownership 等の会社形態は、result-report データの公共性を保つための選択肢として DRI（代表）が判断する。

## 10. 監査・年次透明性レポート

- 年次で公開する透明性レポート: result-report 収集件数・匿名化の適用状況（秘匿セル数等）・スポンサー一覧と金額規模・資金使途・診断ロジックの変更履歴・削除請求の対応件数。
- 匿名化パイプライン（k/l/suppression の適用）は再現可能な形で記述し、可能な範囲でコードを公開する。
- **倫理レビューの枠組み = ODI Data Ethics Canvas**（CC BY-SA 4.0、15ボックス）を年次レビューの骨格に使う。特に本件に効く2ボックス: **Negative effects**（収集・利用・共有が個人/個社を再識別リスク・不当なターゲティング・差別に晒さないか）と **Engaging with people**（訂正・異議申立て・利用停止を合理的手段で行え、周知されているか）。

## 11. 改定手続き

- 本憲章は DRI（代表）が改定権を持つ。実質的改定（収集項目の追加・匿名性閾値の緩和・スポンサー規律の変更等）は変更履歴を残し、consent version を上げる。
- 収集項目を増やす場合は §4 allowlist の改訂 + consent 再取得を要する。
